# System-Operator

Agentic framework for running and operating software systems built by System-Builder. Agents monitor, respond, and maintain operational health; humans handle novel situations and approve high-impact actions.

**Relationship to System-Builder:** System-Builder creates the system and generates the operational artefacts (SLOs, monitoring definitions, deployment topology, runbooks, security posture). System-Operator consumes these artefacts to run the system. When System-Builder reruns (via System-Maintainer's Evolve workflow), Operator receives updated artefacts.

**Relationship to System-Maintainer:** System-Operator runs the system; System-Maintainer changes it. Operator detects problems and either resolves them operationally (restart, scale, failover) or escalates to Maintainer (bug, performance regression requiring code change). Maintainer produces deployable changes and hands them to Operator for rollout. They share the project's operational artefacts but have different read/write patterns: Operator mostly reads, Maintainer updates.

---

## Core Concept

The system is running. The Operator framework:

1. **Observes** — continuously monitors health, performance, and resource utilisation against defined SLOs
2. **Detects** — identifies when something deviates from expected behaviour
3. **Classifies** — determines whether this is an operational issue (handle it) or a code/design issue (escalate it)
4. **Responds** — executes the appropriate runbook or escalation
5. **Verifies** — confirms the response resolved the issue
6. **Reports** — maintains operational history for trend analysis and Maintainer context

The framework is fundamentally **artefact-driven**. Its intelligence lives in the operational artefacts (SLOs, monitoring definitions, runbooks, deployment topology, security posture) generated by System-Builder. Operator agents execute against these artefacts rather than reasoning from first principles. When a runbook doesn't cover a situation, the response is escalation, not improvisation.

## Response Model

The primary classification axis. Determines how much autonomy the Operator has in responding.

| Response level | Definition | Human involvement | Examples |
|----------------|-----------|-------------------|----------|
| **Auto-resolve** | Known problem, known fix, all criteria met | None (notify post-hoc) | Restart crashed pod, auto-scale on load spike, renew certificate |
| **Assisted** | Known problem, but resolution needs judgment or carries risk | Human approves action | Rollback deployment, failover to secondary, increase resource limits beyond threshold |
| **Escalate to Maintainer** | Problem is in code or design, not operations | Maintainer takes ownership | Bug causing errors, performance regression requiring code fix, capacity problem requiring architectural change |
| **Escalate to Human** | Novel situation, no applicable runbook | Human investigates and directs | Unknown failure mode, correlated failures across components, security incident with unclear scope |

### Classification logic

The Operator classifies using this decision tree:

1. **Does a runbook cover this exact scenario?**
   - Yes → follow runbook (auto-resolve or assisted, per runbook's autonomy column)
   - No → go to step 2

2. **Is this an operational issue or a code/design issue?**
   - Operational (infrastructure, resource, external dependency) → attempt resolution with available tools, escalate to human if unsuccessful
   - Code/design (errors in application logic, data corruption, spec violation) → escalate to Maintainer
   - Unclear → escalate to human with diagnostic information

3. **Is this urgent?**
   - Yes (system down, data at risk, SLO budget near exhaustion) → apply immediate mitigation (restart, failover, scale) regardless of classification, then escalate
   - No → classify and route normally

**Key principle:** Operator agents mitigate first, investigate second. If the system is down, restore service before diagnosing root cause. Mitigation actions (restart, scale, failover) are always available regardless of classification.

## Signal Types

What triggers the Operator framework. Unlike Maintainer's signals (which are often human-generated), Operator signals are predominantly machine-generated.

| Signal | Source | Typical response | Urgency |
|--------|--------|-----------------|---------|
| Health check failure | Monitoring system | Auto-resolve (restart) or escalate | Immediate |
| SLO budget burn | SLO tracking | Investigate → mitigate or escalate | High |
| Metric threshold breach | Alerting rules | Per runbook (varies) | Per alert severity |
| Deployment request | System-Maintainer | Execute deployment workflow | Planned |
| Certificate expiry approaching | Scheduled check | Auto-renew | Medium (time-bounded) |
| Backup verification failure | Scheduled check | Alert + investigate | High |
| Security vulnerability detected | Dependency scanning | Assess + escalate to Maintainer | Severity-dependent |
| Cost anomaly | Cost monitoring | Alert + investigate | Medium |
| Infrastructure alert | Cloud provider / K8s | Per runbook | Per severity |
| Capacity threshold approaching | Resource monitoring | Scale or alert | Medium |

### Signal → Maintainer escalation

Some signals start as Operator concerns but become Maintainer signals:

| Operator observes | Operator action | If unresolved → Maintainer signal |
|-------------------|----------------|-----------------------------------|
| Recurring pod crashes | Restart, check resources | Bug report (crash loop after operational causes ruled out) |
| Gradual performance degradation | Scale, check resources | Performance regression (not a resource issue) |
| Error rate increase after deploy | Rollback | Bug report (with deployment correlation data) |
| Dependency failure causing errors | Check dependency, apply fallback | Feature request (improve fallback/resilience) |
| Security scan finds CVE | Assess severity | Dependency vulnerability (for patching) |

The escalation always includes the Operator's diagnostic data — what was observed, what was tried, what the current state is. This saves Maintainer's investigation agent from repeating work.

## Workflows

### Workflow 1: Incident Response

Reactive. Something is wrong, detected by monitoring.

```
Alert → Classify → Mitigate (if urgent) → Diagnose → Resolve or Escalate → Verify → Report
```

**Steps:**

1. **Alert received** — monitoring system fires an alert matching a defined alerting rule. Alert includes: component, severity, metric values, threshold, time.

2. **Classify severity and type** — agent checks:
   - Alert severity (from monitoring definitions)
   - Component criticality (from risk profile)
   - SLO impact (from SLO definitions)
   - Combined assessment determines urgency

3. **Mitigate (if urgent)** — for CRITICAL/HIGH severity: apply immediate mitigation before diagnosis:
   - Service down → restart
   - Resource exhaustion → scale
   - Single instance failure → failover
   - Mitigation actions are pre-authorised in runbooks — no human approval needed for these

4. **Diagnose** — agent follows runbook diagnosis steps:
   - Check the specific runbook section for this alert
   - Execute diagnosis steps in order (check logs, check dependencies, check recent deployments, check resources)
   - Produce: probable cause, affected components, current impact

5. **Resolve or escalate**:
   - **If runbook has resolution for this cause** → apply it (per autonomy column in runbook)
   - **If cause is operational but no runbook** → escalate to human with diagnosis
   - **If cause is code/design** → escalate to Maintainer with diagnosis and operational context
   - **If cause is unclear** → escalate to human with all diagnostic data

6. **Verify** — after resolution:
   - Confirm alert has cleared
   - Check SLO metrics returning to normal
   - Monitor for recurrence (configurable watch period)

7. **Report** — record:
   - Signal, classification, actions taken, outcome
   - Time to detect, time to mitigate, time to resolve
   - Whether escalation was needed and to whom

**Autonomy in incident response:**

| Scenario | Autonomy |
|----------|----------|
| Alert matches runbook, auto-resolve criteria met | Fully automated, notify post-hoc |
| Alert matches runbook, resolution needs approval | Agent proposes action, human approves |
| Alert matches runbook, but runbook says escalate | Generate Maintainer signal with full context |
| Alert doesn't match runbook | Escalate to human with diagnosis |
| CRITICAL severity, any scenario | Mitigate immediately (restart/scale/failover), then classify |

### Workflow 2: Deployment

Triggered by System-Maintainer. Operator manages the rollout and verification.

```
Deployment Request → Pre-flight Checks → Execute Deployment → Smoke Test → Monitor → Confirm or Rollback
```

**Steps:**

1. **Receive deployment request** — from Maintainer. Includes: what changed, which components, deployment notes, test results (already passing from Maintainer's workflow).

2. **Pre-flight checks** — agent verifies:
   - Current system health (don't deploy into an incident)
   - No conflicting deployments in progress
   - Deployment order (from deployment topology) — are prerequisites deployed?
   - Database migrations needed? (from deployment request)
   - Rollback path clear? (previous version available)

3. **Execute deployment** — following deployment topology:
   - Apply database migrations if needed (with pre-migration backup)
   - Deploy components in defined order
   - Progressive rollout if configured (canary → percentage → full)
   - Each stage gates on health checks passing

4. **Smoke test** — after deployment:
   - Run health checks for all deployed components
   - Verify key endpoints respond correctly
   - Check that new functionality is reachable (if test cases provided)

5. **Monitor** — hold period after deployment:
   - Watch error rates, latency, resource usage
   - Compare against pre-deployment baseline
   - Duration: configurable (default: 15 minutes for standard, 1 hour for high-risk)
   - SLO budget burn rate must not increase

6. **Confirm or rollback**:
   - **If monitoring is clean** → confirm deployment, notify Maintainer of success
   - **If anomalies detected** → assess severity:
     - Minor: alert human, continue monitoring
     - Significant: propose rollback, human approves
     - Critical (system down): auto-rollback, notify human and Maintainer

**Rollback procedure:**
- Revert to previous container images (deployment topology defines the order)
- If database migration was applied: forward-fix preferred (rollback migrations are risky). Escalate to Maintainer if forward-fix is needed.
- Report: what was deployed, what went wrong, what was rolled back

### Workflow 3: Routine Operations

Scheduled tasks that maintain system health. Proactive rather than reactive.

```
Schedule Trigger → Execute Task → Verify → Report (or Escalate if failed)
```

**Tasks:**

| Task | Frequency | Auto-resolve? | Failure action |
|------|-----------|---------------|----------------|
| Certificate renewal | Check daily, renew at 30 days before expiry | Yes (cert-manager) | Alert human if auto-renewal fails |
| Backup verification | Daily | Yes (automated restore test) | CRITICAL alert if verification fails |
| Dependency vulnerability scan | Daily | No (escalate to Maintainer) | Severity-based: CRITICAL → immediate escalation, others → batched |
| Database maintenance | Weekly | Yes (vacuum, reindex) | Alert if maintenance fails or takes too long |
| Log rotation / cleanup | Daily | Yes | Alert if disk usage still high after cleanup |
| Secret rotation | Per schedule in security posture | Yes where automated | Alert before expiry if manual rotation needed |
| SLO report generation | Weekly / monthly | Yes | Alert if report generation fails |
| Cost report generation | Monthly | Yes | Alert if anomalies detected |

**Scheduled task agent:**
- Reads the task schedule and executes at the defined frequency
- Each task has explicit success criteria and failure actions
- Failed tasks are retried once, then escalated (to human for operational tasks, to Maintainer for code-adjacent tasks like dependency updates)
- All task executions are logged for audit

### Workflow 4: Capacity Management

Ongoing. Manages resource allocation and scaling decisions.

```
Monitor Resources → Assess Against Thresholds → Scale or Alert → Verify → Report
```

**Dimensions:**

| Dimension | Metric | Scale trigger | Scale action | Limit |
|-----------|--------|---------------|-------------|-------|
| Compute (per component) | CPU utilisation | > threshold (from deployment topology) | Add replicas | Max replicas (from deployment topology) |
| Memory (per component) | Memory utilisation | > 80% sustained | Increase limits | Alert if near node capacity |
| Storage | Disk utilisation | > 80% | Expand volume or clean up | Alert if expansion not possible |
| Queue depth | Messages pending | > threshold (from monitoring definitions) | Scale consumers | Max consumers (from deployment topology) |

**Scaling autonomy:**
- **Within defined limits** (min/max replicas, resource bounds in deployment topology): fully automated
- **Beyond defined limits** (needs more capacity than configured maximum): alert human with recommendation
- **Cost implications** (scaling would increase monthly cost by >X%): alert human with cost projection

**Capacity forecasting** (stretch goal):
- Track resource usage trends over time
- Project when current capacity will be insufficient
- Generate early warnings: "At current growth rate, consumer-api will hit max replicas in ~30 days"
- These projections become Maintainer signals if they require architectural change

## Artefact Consumption

System-Operator is driven by the operational artefacts generated by System-Builder. Here's how each artefact maps to Operator workflows:

| Artefact | Used by | How |
|----------|---------|-----|
| SLO Definitions | Incident Response, Capacity, Reporting | Determines severity, budget tracking, success criteria |
| Monitoring Definitions | Incident Response, Deployment | Defines what to watch, when to alert, health check criteria |
| Deployment Topology | Deployment, Capacity | Deployment order, scaling limits, resource profiles |
| Runbooks | Incident Response, Routine Ops | Decision trees for diagnosis and resolution |
| Security Posture | Routine Ops (secret rotation, cert renewal) | Rotation schedules, security monitoring rules |
| Component Map | Incident Response (blast radius) | Understanding which components are affected by an incident |
| Risk Profile | Incident Response (severity assessment) | Component criticality informs response urgency |

**Artefacts the Operator generates** (operational data, not design docs):

| Artefact | Contents | Consumer |
|----------|----------|----------|
| Incident log | Timestamped record of all incidents, responses, outcomes | Human review, Maintainer context, trend analysis |
| Deployment log | Record of all deployments, results, rollbacks | Maintainer (deployment correlation), human audit |
| SLO reports | Periodic compliance reports, budget status, trends | Human review, product decisions |
| Cost reports | Resource usage, cost trends, anomalies | Human review, capacity planning |
| Capacity forecasts | Resource trend projections | Human planning, Maintainer (architectural signals) |

## Autonomy Model

Operator has higher baseline autonomy than Maintainer because its actions are more bounded and reversible.

### Tier 1: Fully automated (notify post-hoc)
- Pod restart on crash
- Auto-scale within defined limits
- Certificate auto-renewal
- Log rotation and cleanup
- Health check-driven restarts
- Criteria: action is in runbook with auto-resolve, within resource limits, no data risk

### Tier 2: Automated with immediate notification
- Deployment rollback on critical failure
- Failover to secondary
- Emergency scaling beyond normal limits
- Criteria: urgent mitigation, reversible action, time-critical (waiting for human would worsen impact)

### Tier 3: Propose action, wait for approval
- Standard deployment execution
- Rollback on non-critical anomalies
- Resource limit increases beyond defined thresholds
- Infrastructure changes (e.g., adding a read replica)
- Criteria: non-urgent, or significant cost/infrastructure implications

### Tier 4: Escalate (Operator cannot resolve)
- Novel failure modes (no matching runbook)
- Code or design issues detected (→ Maintainer)
- Security incidents with unclear scope (→ Human + Maintainer)
- Correlated failures suggesting systemic issue (→ Human)
- Criteria: outside Operator's action space, requires investigation or code change

**Default posture:** Start conservative. Tier 1 actions should be limited to well-understood, low-risk, reversible operations. Expand the auto-resolve set as confidence grows.

## Communication Patterns

### Operator → Human
- **Post-hoc notifications**: Tier 1 actions, resolved incidents, completed routine tasks
- **Approval requests**: Tier 3 actions, proposed responses needing judgment
- **Escalations**: Tier 4 situations, novel failures, security concerns
- **Reports**: SLO compliance, cost trends, capacity forecasts (periodic)

### Operator → Maintainer
- **Bug signals**: recurring crashes, error patterns after operational causes ruled out
- **Performance signals**: degradation not explained by load or resources
- **Dependency signals**: CVE detections, dependency health issues needing code changes
- **Capacity signals**: architectural limits being approached (need design change, not just more resources)
- **Deployment feedback**: rollback occurred (Maintainer's change caused issues), deployment anomalies

All escalations include: what was observed, what was tried, current system state, relevant metrics/logs.

### Maintainer → Operator
- **Deployment requests**: deploy this change, here's what changed and what to watch for
- **Artefact updates**: updated runbooks, changed SLOs, new monitoring rules
- **Watch requests**: "we deployed a fix for X, monitor Y metric for regression"

### Human → Operator
- **Directives**: "scale component X to N replicas", "execute runbook section Y"
- **Approvals**: "proceed with rollback", "approve resource increase"
- **Overrides**: "ignore this alert (false positive)", "hold deployment"

## Operational State

Unlike Maintainer (which tracks design doc state), Operator tracks runtime state.

**What Operator tracks:**
- Current deployment versions (per component, per environment)
- Active incidents (open, mitigating, resolved)
- SLO budget status (remaining budget per SLO per window)
- Recent deployment history (for correlation with incidents)
- Scheduled task status (last run, next run, last result)
- Scaling state (current replicas, recent scaling events)

**Where this lives:** TBD — could be a state file in the project repo, a database, or an external system. The framework is agnostic; it needs a read/write store for operational state that persists across agent invocations.

## Design Decisions

Resolved from the original open questions during architecture design.

### State persistence

The framework defines an abstract state interface (`State.read`/`State.write`) — storage implementation is project-specific. Simple systems use structured files in `operations/state/`; production systems use a database or external state service. The framework is agnostic. See ARCHITECTURE.md § State Management.

### Multi-environment

Separate Operator instance per environment. Shared agent prompts, but each environment gets its own artefact paths, tool implementations, and state store. Staging may have relaxed SLO targets. This avoids environment-routing complexity inside the Operator.

### Alert deduplication and grouping

The Dispatcher uses the Component Map's dependency graph to group related alerts. When an alert fires for component A, check whether A's dependencies have active incidents — if component B (which A depends on) already has an active incident, correlate A's alert into B's incident rather than creating a new one. Root-cause-first grouping.

**[REFINE DURING IMPLEMENTATION]** Edge cases need attention: correlated failures not in the dependency chain, cascading failures where root cause isn't the first alert, and appropriate time windows for correlation.

### Observability bootstrapping

First deployment enters a "baseline collection" phase: monitoring is active but alerting uses absolute thresholds only (e.g., "service down", "error rate > 50%") — no anomaly-based or trend-based alerts. After a configurable period of stable operation (default: 7 days), baselines are established from collected metrics and normal alerting thresholds activate.

**[REFINE DURING IMPLEMENTATION]** What constitutes "stable operation"? Transition from relaxed to normal thresholds (gradual vs cliff). Handling systems that don't stabilise within the baseline window.

### Cross-system operations

One Operator per system. Each system has its own artefacts, state, and tool implementations. If systems share infrastructure, infrastructure-level alerts are handled by whichever system's Operator owns that infrastructure component in its Component Map.

### Runbook evolution

Operator encounters novel scenario → escalates to Human → Human resolves → Operator records the resolution in the Incident Log (observations, actions, outcome) → Operator signals Maintainer with "runbook gap" (pointing to incident log entry) → Maintainer authors new runbook section → Artefact Update signal back to Operator. Key principle: Operator provides the raw material (incident data), Maintainer does the authoring. Operator never writes runbooks.

## What's NOT in System-Operator

- **Code changes** (bug fixes, features, refactoring) → System-Maintainer
- **Design decisions** (architecture, component design) → System-Maintainer → System-Builder
- **Initial system creation** → System-Builder
- **Strategic product decisions** → Human
- **Root cause analysis requiring code investigation** → System-Maintainer (Operator provides operational context)
- **Dependency patching** (code change) → System-Maintainer (Operator detects the CVE, Maintainer patches it)
